What it does

Keep workflow contracts close to the automation they describe.

Declarations live under .github/actionspec, can be discovered by workflow name, and can be validated against the same JSON payloads your workflows actually produce.

Why it helps

  • Catch drift before a workflow change becomes operational breakage.
  • Run the same validation surface locally, in CI, and through the action.
  • Adopt incrementally with one workflow and one real payload first.

Three ways in

Use the interface that fits the way your team already works.

Discover declarations

Scan a repository and map workflow files to contract declarations.

github-actionspec discover --repo /path/to/repo

Validate from the repo root

Resolve the matching declaration for a workflow and validate one or many payloads.

github-actionspec validate-repo \
  --repo /path/to/repo \
  --workflow build-infrastructure.yml \
  --actual .github/actionspec-artifacts/**/*.json

Use the GitHub Action

Drop the same validation flow into CI with minimal wrapper logic.

- uses: v4lproik/github-actionspec-rs@main
  with:
    repo: .
    workflow: build-infrastructure.yml
    actual: .github/actionspec-artifacts/**/*.json

Contract shape

Readable enough to inspect, structured enough to enforce.

The declaration layer stays close to the workflow domain instead of hiding validation rules behind shell conditionals.

Declaration

package actionspec

workflow: "build-infrastructure.yml"

run: #Declaration.run & {
  workflow: workflow
  jobs: {
    sample: {
      result: "success"

  }
}
