Keep contracts in the repo. Validate the payloads your workflows actually emit.

What it does

Keep contracts next to the workflows they describe.

Store declarations under .github/actionspec, resolve them by workflow name, and validate the JSON payloads your jobs already produce.

discover validate-repo github action

Why it helps

  • Catch workflow drift before it breaks automation.
  • Use the same validation flow locally and in CI.
  • Start with one workflow and one payload.

Three ways in

Run it where you need it.

Inspect

Discover declarations

Scan a repository and map workflows to declarations.

github-actionspec discover --repo /path/to/repo

Most direct

Validate from the repo root

Resolve one workflow and validate one or many payloads.

github-actionspec validate-repo \
  --repo /path/to/repo \
  --workflow build-infrastructure.yml \
  --actual .github/actionspec-artifacts/**/*.json

Automate

Use the GitHub Action

Run the same check in CI with minimal wrapper logic.

- uses: v4lproik/github-actionspec-rs@main
  with:
    repo: .
    workflow: build-infrastructure.yml
    actual: .github/actionspec-artifacts/**/*.json

Contract shape

Readable enough to review. Strict enough to enforce.

Keep the validation rules in CUE instead of spreading them across shell checks.

Declaration

package actionspec

workflow: "build-infrastructure.yml"

run: #Declaration.run & {
  workflow: workflow
  jobs: {
    sample: {
      result: "success"
    }
  }
}
